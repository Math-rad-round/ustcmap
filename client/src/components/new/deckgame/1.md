我们正在开发一款校园成长主题的 Deck Building（牌组构筑）桌游电子版，计划先开发 Web 版本，再考虑移植实体桌游。游戏定位为轻中度德式策略游戏，强调长期规划而非随机性，支持单人与多人，多人互动以合作、竞争和资源争夺为主，不采用直接攻击。玩家扮演大学新生，在 8 个学期中不断学习成长，每个学期分为学习生活阶段和考试阶段。学习阶段通过事件获得新课程、升级课程、删除低效课程、获得成果等奖励，不断优化自己的课程牌组；考试阶段将整个牌组洗混，根据当学期考试要求抽取固定数量（目前测试为4~6张）的课程牌，统一统计知识符号并结算所有课程效果，最后根据考试得分获得 Achievement Points（成就点），8 个学期结束后成就点最高者获胜。

课程牌是游戏的核心。每张课程牌包含两部分：知识符号（目前为 Theory、Practice、Social 三种）以及考试效果。初始牌组为 Theory×3、Practice×3、Social×3、Blank×1，共10张。课程效果不是打出时立即生效，而是在考试统一结算，例如“Theory≥3，则Score+6”“每个Practice，Score+1”“每组Practice+Social，Score+2”等，因此课程本质上更像一条规则（Rule），而不是传统卡牌游戏中的动作（Action）。考试要求（Exam）每学期随机变化，可以修改知识权重、抽牌数量、评分方式等，例如理论考试提高 Theory 收益，工程实践提高 Practice 收益，以鼓励玩家不断调整成长方向。成果（Achievement）不进入牌组，而是永久放置，例如奖学金、科研项目、社团干部等，可以提供长期能力、特殊规则或额外成就点。学习阶段的事件遵循“选择→成本→奖励”的设计，而不是抽到即获得奖励，多人模式中的合作事件允许玩家共同完成大型项目并按贡献分配奖励。

电子版采用 React Class Component + React Router，作为现有网站中的一个模块（类似 Map 页面），而不是独立 React 项目。目前技术栈为 React Class Component、JavaScript ES6，未来可能迁移到 Electron，但优先开发 Web 版本。整个电子版采用“Engine + UI 分离”的架构，即 React(UI) → GameEngine → GameState → JSON Database。React 只负责显示界面和响应用户点击，不参与任何游戏规则计算；GameEngine 负责所有游戏逻辑；GameState 保存整个游戏状态；JSON 保存所有课程、事件、考试、成果等数据。

整个项目采用 Command Driven（命令驱动）而不是 React 直接修改状态。React 永远不能直接执行诸如 player.score++、deck.push(card) 等操作，而只能向 Engine 发送命令，例如 START_SEMESTER、REMOVE_CARD、UPGRADE_CARD、GAIN_CARD、CHOOSE_EVENT、FINISH_SEMESTER 等。GameEngine 接收命令、验证是否合法、修改 GameState，然后 React 根据新的 GameState 重新 render。以后 AI、玩家、联机、录像、回放、Undo 都统一通过发送 Command 操作 Engine，而不是直接修改数据。需要注意的是，GameEngine 并不是后台持续运行的线程，它只是普通的 JavaScript 对象，浏览器仍然只有一个主线程，流程实际上是“React 收到点击 → 调用 Engine 计算几毫秒 → Engine 返回新的 GameState → React 重新渲染”。

整个游戏采用 Data Driven（数据驱动）设计。课程、事件、考试、成果全部使用 JSON 配置，而不是每张牌对应一个 JavaScript 类。新增一张课程应尽量只增加一段 JSON，而不是修改 Engine。Engine 负责解释这些 JSON。整个项目不会存在 HighMath.js、Physics.js、Chemistry.js 这种“一张牌一个类”的设计，而是采用 Rule Engine + Effect System。课程只是若干规则的组合，例如 Condition（条件）+ Action（动作），例如“Theory≥3 → Score+6”“Always → ForEach Practice → Score+1”。Engine 负责解释这些规则，而不是写 if(card.name=="高中数学") 这样的代码。整个项目只维护几十种通用 Effect，例如 Threshold、Count、Pair、Compare、Bonus、Repeat、Choice、Upgrade、Remove、Draw 等，以后数百张课程牌都只是这些 Effect 的组合，而不是不断增加新的代码。

目录规划为 deckgame/ui（React 界面）、engine（游戏逻辑）、model（Player、Deck、Course 等数据对象）、database（读取 JSON）、data（全部 JSON 配置）、ai（AI）、util（工具函数）。Engine 是整个项目最稳定的核心，React 是可以替换的外壳，内容（课程、事件、考试、成果）则不断通过 JSON 扩展。

目前准备先完成一个最小可运行 Demo（MVP），只有 10 张牌（Theory×5、Practice×5），点击“开始考试”后由 Engine 洗牌、抽取5张、统计 Theory 数量和 Practice 数量，并计算 Score = Theory × Practice，React 仅负责显示抽到的牌、Theory 数量、Practice 数量和最终 Score。这个 Demo 的目的不是完成游戏，而是验证整个架构是否正确，即 React → GameEngine → GameState → React Render 的完整运行流程。整个项目的长期目标不是开发一个简单网页游戏，而是构建一个可扩展的桌游引擎，通过新增 JSON 和少量新的 Effect 类型不断扩展内容，而无需重构底层代码，未来能够支持数百张课程牌、事件、成果，以及 AI、联机、录像、回放和 Mod。
