# 多人游戏功能规划

## 核心定位

多人模式不是多人共同操作一个学生，而是每名玩家操作一个不同的学生。每个学生独立成长、独立考试、独立获得成就点；所有玩家在同一个学期节奏中竞争，最终成就点最高者获胜。

多人模式的关键词是：独立学生、共享考试池、合作事件、独立温习、独立考试、分数排名、共享目标竞争。

## 基础结构

一个多人房间包含 n 名玩家，每名玩家对应一个学生状态。

每个学生拥有自己的：

- 课程牌堆。
- 拥有课程库。
- 事件手牌。
- 项目区。
- 成果区。
- 温习值。
- 临阵磨枪、重考、重抽考试等个人机会。
- 成就点。
- 考试历史。

房间共享：

- 当前学期。
- 当前阶段。
- 本学期考试候选池。
- 即时目标池。
- 终局目标池。
- 合作事件或合作项目的公共结算流程。
- 操作日志。

## 学期流程

每个学期仍然分为：

1. 考试选择阶段。
2. 事件阶段。
3. 温习阶段。
4. 考试阶段。
5. 排名与目标结算。

所有玩家处在同一个大阶段里，但很多操作是各自独立完成。

## 考试选择阶段

### 共享考试池

每学期开始时，房间从考试牌堆中抽出 n 张考试卡，形成共享考试池。n 是玩家人数。

所有玩家都从这同一组考试卡中独立选择一张作为本学期考试。

### 是否允许选择同一场考试

允许多人选择同一场考试。

如果某场考试被多名玩家选择，则这些玩家除了参与全房间总排名外，还会在这场考试内部进行额外排名修正：

- 同考试第一名：额外 +1 成就点。
- 同考试最后一名：额外 -1 成就点。
- 只有 1 人选择某场考试时，不触发这个同考试修正。

这个规则的作用是：允许抱团选择同一考试，但会产生小范围竞争压力。

## 事件阶段

### 个人行动次数

每名玩家每个事件阶段各有 2 次行动。

单人事件只消耗执行者自己的 1 次行动。

### 现有事件

当前 events.json 中已有事件默认都是单人事件。它们只影响执行者，或者由执行者选择效果归属。

也就是说：谁获得奖励，就给谁；谁获得课程，就加入谁的课程牌堆；谁获得成果，就进入谁的成果区。

成果卡没有共享概念。

### 合作事件

后续会设计新的合作事件，或者给部分事件增加合作选项。

合作事件的基本规则：

- 只有参与合作的玩家消耗行动。
- 合作事件消耗所有参与者的 1 次行动。
- 未参与者不消耗行动。
- 合作事件结算时，所有参与者都执行效果、都获得对应奖励。
- 如果奖励需要选择归属，则按照事件文本指定或由参与者各自选择。

示例：

```text
合作讨论
参与人数：2-4
消耗：每名参与者 1 次事件行动
效果：每名参与者获得 2 温习值；如果参与者中有人拥有科研成果，每名参与者额外获得 1 成就点。
```

### 合作支付

合作事件可以设计为多人共同支付费用，例如每名参与者弃置 1 张事件，或者合计弃置指定数量事件。

建议第一版先使用简单格式：每名参与者支付相同费用。后续再支持“合计支付”。

## 温习阶段

温习阶段完全独立。

每名玩家查看和调整自己的课程牌堆。玩家之间不能直接操作他人的牌堆。

所有玩家都确认温习结束后，房间进入考试阶段。

如果做实时同步，其他玩家可以看到“某玩家正在温习 / 已完成温习”，但不暴露其牌堆顺序。

## 考试阶段

每名玩家独立考试。

每个学生使用自己的：

- 课程牌堆。
- 当前选择的考试卡。
- 温习调整结果。
- 成果区。
- 项目进度。
- 个人补救机会。

考试分数独立计算。

### 多人考试档位奖励

多人模式仍然保留考试档位奖励，但比单人模式低。

四档奖励分别比单人减少：

```text
- 第一档：-1
- 第二档：-1
- 第三档：-2
- 第四档：-2
```

例如普通单人奖励是：

```text
1 / 4 / 6 / 8
```

多人普通奖励变为：

```text
0 / 3 / 4 / 6
```

困难、简单奖励也按同样规则降低，但最低不低于 0。

### 未达标惩罚

未达到本次考试最低线时，仍然扣除 5 成就点。

排名奖励不能完全抵消挂科压力。

## 分数排名奖励

所有玩家完成考试后，按本学期考试总分排名。

基础排名奖励：

```text
第 1 名：+3 成就点
第 2 名：+2 成就点
第 3 名：+1 成就点
其他名次：+0 成就点
```

### 并列排名

并列时，把被占用的名次奖励合计，然后下取整平分。

示例：

```text
两人并列第一：
第 1 名和第 2 名奖励合计 3 + 2 = 5
5 / 2 = 2.5，下取整
两人各 +2
下一名按第 3 名，获得 +1
```

```text
三人并列第一：
第 1、2、3 名奖励合计 3 + 2 + 1 = 6
6 / 3 = 2
三人各 +2
下一名无排名奖励
```

```text
两人并列第二：
第 2、3 名奖励合计 2 + 1 = 3
3 / 2 = 1.5，下取整
两人各 +1
```

### 同考试内部排名修正

如果同一场考试被多名玩家选择，则这些玩家之间额外比较一次该考试分数：

- 该考试第一名：+1 成就点。
- 该考试最后一名：-1 成就点。

如果同考试内部出现并列，可以沿用并列平分思路：

- 并列第一都视为第一，获得 +1。
- 并列最后都视为最后，受到 -1。
- 如果所有选择该考试的玩家完全同分，则同时是第一和最后，+1 与 -1 抵消，最终 0。

## 目标系统

### 即时目标

多人模式下，即时目标是共享竞争目标。

所有玩家争同一批即时目标，谁先达成谁获得奖励。

多人模式下所有目标都没有实现期限，不会在第 3 / 4 / 5 学期移除。

也就是说，原本单人模式中的 deadline 在多人模式中禁用。

### 终局目标

终局目标也是共享竞争目标。

终局目标的含义是：“某条件最高的人获得奖励”。

例如：

```text
理论大师：最终理论符号最高的玩家获得奖励。
项目达人：最终拥有和完成项目总数最高的玩家获得奖励。
积累者：最终成果数量最高的玩家获得奖励。
绩点优秀：最终考试成就或总表现最高的玩家获得奖励。
```

如果终局目标出现并列，可以有两种方案：

- 方案 A：并列最高者都获得完整奖励。
- 方案 B：并列最高者平分奖励，下取整。

建议第一版使用方案 A，因为更直观，也更不容易让玩家觉得被系统亏待。

## 项目系统

### 个人项目

现有项目卡默认是个人项目。

谁获得项目，项目就进入谁的项目区。

谁完成项目，谁获得奖励。

成果卡没有共享概念。项目奖励如果给成果，则成果进入完成者的成果区。

### 合作项目

后续可以设计新的合作项目卡。

合作项目允许多名玩家共同推进，并在完成时让参与者都获得奖励。

合作项目建议记录贡献：

```js
contributors: {
  [playerId]: progressAmount
}
```

合作项目完成时，可以按项目文本结算：

- 所有贡献者都获得奖励。
- 贡献最高者额外获得奖励。
- 每有 1 点贡献获得对应收益。
- 项目负责人获得成果，其他贡献者获得成就点或温习值。

第一版可以先做简单合作项目：所有参与者都获得同一奖励。

## 胜负条件

游戏结束时，每名玩家结算自己的最终成就点。

成就点最高者获胜。

如果最高成就点并列，则可以使用以下加赛比较：

1. 总考试分更高者胜。
2. 完成项目更多者胜。
3. 成果更多者胜。
4. 仍然并列则共同胜利。

## 状态结构建议

多人房间状态可以拆成房间状态和玩家状态。

```js
{
  roomId,
  roomCode,
  semester,
  phase,
  version,
  playerOrder: [playerId],
  examCandidates: [testCard],
  sharedImmediateGoals: [],
  sharedFinalGoals: [],
  players: {
    [playerId]: {
      userId,
      name,
      studentName,
      state: GameState,
      selectedTestId,
      eventActionsUsed,
      ready: false,
      connected: true,
      examResult: null
    }
  },
  logs: []
}
```

这里的 `state: GameState` 可以尽量复用当前单人 GameState。多人控制器负责协调阶段、考试池、排名和共享目标。

## 引擎改造建议

不要立刻把当前 GameEngine 改成多人引擎。

建议新增一个多人控制层：

```text
MultiplayerGameEngine
  - 管理房间阶段
  - 管理玩家列表
  - 给每名玩家持有一个 GameEngine / GameState
  - 分发 command 到对应玩家引擎
  - 处理合作事件
  - 处理排名奖励
  - 处理共享目标
```

单人 GameEngine 继续负责：

- 抽牌。
- 事件结算。
- 温习。
- 考试计分。
- 项目和成果。

多人控制层负责：

- 谁能操作。
- 谁已经确认。
- 哪些玩家参与合作事件。
- 多人考试池。
- 考试排名奖励。
- 同考试内部修正。
- 共享目标竞争。

这样可以避免把单人模式搅乱。

## 后端设计建议

多人模式必须由后端保存权威状态。

前端不应该提交完整 gamedata 来覆盖房间状态，而应该提交 command。

推荐 API：

```text
POST /api/game/room/create
POST /api/game/room/join
POST /api/game/room/list
POST /api/game/room/load
POST /api/game/room/command
POST /api/game/room/leave
```

command 示例：

```js
post('/api/game/room/command', {
  roomId,
  version,
  playerId,
  command: { type: 'PLAY_EVENT', eventIndex: 0, optionIndex: 1 }
})
```

合作事件 command 示例：

```js
post('/api/game/room/command', {
  roomId,
  version,
  playerId,
  command: {
    type: 'PLAY_COOP_EVENT',
    eventIndex: 0,
    optionIndex: 1,
    participantIds: ['p1', 'p2']
  }
})
```

后端需要校验：

- 玩家是否在房间中。
- 是否处于正确阶段。
- 玩家是否还有行动次数。
- 合作参与者是否都确认参与。
- command 的 version 是否匹配。

## 前端 UI 需要新增

### 主菜单

- 新建多人房间。
- 输入房间码加入。
- 我的多人房间列表。
- 继续最近多人房间。

### 房间大厅

- 玩家列表。
- 准备、取消准备。
- 房主开始游戏。
- 复制邀请链接。
- 玩家选择学生名称或头像。

### 游戏内布局

建议采用“自己的学生为主，其他学生为侧栏”的布局。

主区域显示：

- 我的课程牌堆和手牌。
- 我的项目和成果。
- 我的温习、考试、事件操作。

侧栏显示其他玩家摘要：

- 当前成就点。
- 当前考试选择。
- 是否已完成本阶段。
- 本学期考试分数。
- 项目数量、成果数量。

公共区域显示：

- 共享考试候选。
- 共享目标。
- 排名面板。
- 房间日志。

### 合作交互

合作事件需要专门 UI：

- 选择合作事件。
- 邀请参与者。
- 参与者确认。
- 显示每名参与者需要支付的费用。
- 所有人确认后执行。

第一版可以简化为：发起者选择参与者后，系统只允许已经点“同意参与”的玩家加入结算。

## 同步策略

第一版建议轮询，不急着 WebSocket。

- 每 2 秒拉取房间状态。
- 玩家提交 command 后立即刷新。
- 使用 version 避免并发冲突。

后续再做 WebSocket 推送。

## 最小可行版本

第一版多人模式可以只做这些：

1. 房间系统。
2. 每名玩家一个独立 GameState。
3. 共享抽 n 张考试卡。
4. 每名玩家独立选择考试。
5. 每名玩家独立事件阶段，各 2 次行动。
6. 暂不实现合作事件，只保留单人事件。
7. 每名玩家独立温习和考试。
8. 考试档位奖励按多人规则降低。
9. 考试后结算全房间排名奖励。
10. 支持同考试第一 +1、最后 -1。
11. 即时目标共享竞争，无期限。
12. 终局目标按最高者结算。
13. 第 8 学期结束后按成就点排名，最高者获胜。

合作事件和合作项目可以作为第二版。

## 后续增强

- 合作事件。
- 合作项目。
- 玩家之间交易事件牌或项目推进。
- 玩家专属能力。
- 院系、专业方向差异。
- WebSocket 实时同步。
- 多人 AI 毕业总结。
- 回放系统。

## AI 总结接入思路

多人最终总结可以把以下内容发给 AI：

- 房间玩家列表。
- 每名玩家最终成就点。
- 每名玩家考试历史。
- 每名玩家完成项目和成果。
- 共享目标归属。
- 每学期排名。
- 合作事件和合作项目记录。
- 最终胜负结果。

调用方式仍然沿用：

```js
post('/ai/parse', { text: parseText })
```

AI 输出可以包括：

- 总体毕业故事。
- 冠军玩家评价。
- 每名玩家路线总结。
- 关键竞争回合。
- 合作事件亮点。
- 下一局建议。
